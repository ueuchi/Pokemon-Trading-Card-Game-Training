## 1. プロジェクト概要

ポケモンカードゲームのルールに基づいた対戦ゲームを自作する。  
最終目標は **CPU対戦** および **友達との2人対戦（リアルタイム）** が動作するWebアプリケーション。

### MVP（最小実用プロダクト）の定義

- カードをデータベースに登録できる
- 登録したカードでデッキを作成できる
- 作成したデッキでCPUと対戦できる
- 作成したデッキで友達とリアルタイム対戦できる

## 3. 後回しにする機能（スコープ外）

以下の機能は現時点では **実装しない**。ゲームの複雑性を抑え、コアな対戦体験の完成を優先する。

### Phase 8以前では実装しない

| 機能 | 理由 |
|------|------|
| 特殊状態の効果処理 | ゲームエンジンの複雑性が大幅に増すため Phase 9 以降に後回し |
| ルールを持つポケモン（ex・Vなど）のサイド複数枚取得 | カードデータが安定してから対応 |
| 特殊な特性によるバトル場以外からの場への登場 | 例外処理が多いため後回し |

### 全フェーズを通じてスコープ外

| 機能 | 理由 |
|------|------|
| ユーザー認証・ログイン機能 | 初期はローカル・友達間での利用を想定 |
| ランキング・マッチメイキング | 対象規模が大きくなってから検討 |
| カード画像の外部API連携（PokeAPI等） | 著作権・安定性の観点から自前画像テンプレートで対応 |
| サウンドエフェクト・BGM | UI/UX改善フェーズ以降で検討 |
| スマホネイティブアプリ（iOS/Android） | WebアプリのレスポンシブでまずはカバーしPhase 10以降で検討 |
| 課金・決済機能 | スコープ外 |

## 4. 各Phaseの完了条件

| Phase | 完了条件 |
|-------|---------|
| **Phase 0** | すべての仕様書（game-rules.md / scope.md / database-schema.md / card-image-strategy.md）が完成し、内容を確認・合意できた状態 |
| **Phase 1** | Pythonからデータベースに接続・CRUD操作ができ、すべてのテストがパスする |
| **Phase 2** | DBにカードを追加 → API経由で取得 → Angularで一覧表示まで即反映される |
| **Phase 3** | PythonだけでCLIベースの2プレイヤー対戦が最初から勝敗判定まで完走する |
| **Phase 4** | CPUを相手に1試合完走でき、勝敗が正しく判定される |
| **Phase 5** | UIからカードを追加・編集・削除でき、変更がDBに即反映される |
| **Phase 6** | UIからデッキを作成・保存でき、60枚・4枚制限が正しく機能する |
| **Phase 7** | ブラウザ上でCPUと1試合完走でき、すべての基本操作がUIから行える |
| **Phase 8** | 異なる2ブラウザからルームを作成・参加し、リアルタイムで1試合完走できる |
| **Phase 9** | 進化システムと特殊状態（どく・やけど等）がゲーム内で正しく機能する |
| **Phase 10** | アニメーション・レスポンシブ・ダークモードが実装され、UXが大幅に改善されている |
| **Phase 11** | 複雑なトレーナー効果・デッキテンプレート・対戦履歴が使用できる |
